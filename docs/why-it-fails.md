# 为什么 prompt 压不住中文 AI 腔

这个项目最初的假设是「列清楚禁止表达，模型就不会用」。实际效果远远低于预期：模型会一边说"对，规则禁了这个句式"，一边在下一段继续用同样的句式。

这个文档解释这件事的根本原因，以及对应的设计取舍。

## 1. 这是模型权重层面的问题

中文 AI 腔不是模型"不知道这些是口癖"，而是 RLHF 训练把这些模式焊死在了权重里。具体机制：

- **typicality bias**：标注员系统性偏好"典型"、"四平八稳"、"听起来很会讲话"的文本
- 这种偏好被编码进 reward model
- 经过 PPO 训练后，模型在生成时对这些模式有强先验概率
- 单纯靠 prompt 给一个反向指令，只能小幅扰动概率分布，压不住几十个百分点的先验

研究数据：在 TruthfulQA 上，40-79% 的问题在多次采样里只产生一个语义簇——这就是中文输出反复出现「稳稳接住你」「如果你愿意」的原因。模式坍塌已经发生在权重层。

## 2. 负面提示有 priming 效应

直觉上「告诉模型不要说 X」应该能压住 X。实际上会激活 X 的表征。

研究测量结果：

| 场景 | 目标 token 概率下降 |
|---|---|
| 负面指令成功 case | 22.8 个百分点 |
| 负面指令失败 case | 5.2 个百分点 |

4.4 倍的差距说明，负面指令在很多情况下根本压不住目标 token，反而通过提及它强化了关联。

这意味着我们在 SKILL.md 里详细列出 60+ 条被禁表达的做法，**本身就在持续给模型 priming**。

## 3. 项目数量有硬上限

跨 GPT-4、GPT-3.5、LLaMA 的实验结果：每个 prompt 里有效的硬禁令上限是 **3-5 条**。超过这个数，注意力被稀释，每条禁令的实际抑制力都会下降。

我们之前的 9 类禁令、60+ 条目，在工程上是反优化。

## 4. 自回归生成没有回溯能力

模型生成是逐 token 的。一旦写出了「这不是」，后面接「而是」的概率就被强烈牵引——因为这是训练数据里最高频的延续。等到一句写完，已经回不去了。

规则文本只在生成开始前发挥作用。生成过程中，规则的影响力随上下文长度衰减。

## 5. 训练目标和我们的目标错位

模型的训练目标是「让标注员喜欢」。中文标注员（特别是商业场景）历史上偏好：

- 礼貌客服腔
- 心理咨询式安抚
- 商务黑话
- 排比对举
- 升华总结

这些模式得到了系统性奖励。我们的项目目标恰好是反过来——让输出像一个**不在乎被喜欢**的人写的。这是在跟训练目标对抗。

## 我们能做什么

### 有效的做法

1. **聚焦前 5 条硬禁令**，而不是列 60 条
2. **用 FORBIDDEN/NEVER 这类高显著性词汇**，比"避免"、"尽量不要"有效 50-67%
3. **正向锚定 + 负向禁令组合**：不只是说"不要 X"，同时给出"应该写成 Y"的具体范本
4. **输出前自检步骤**：在 SKILL 里强制要求"发送前重扫一遍最后一段"
5. **配合 lint 的多轮重写**：生成 → lint → 重写。规则不能保证一次成功，工程流程兜底
6. **提供 verbalized sampling 出口**：让模型生成多个候选并自评，能部分缓解模式坍塌

### 没用或反作用的做法

1. **超过 5 条的硬禁令清单**——稀释注意力
2. **在 always-on rule 里详细罗列被禁表达**——priming 效应
3. **只说"不要 X"不给 Y 的替代**——模型不知道往哪走
4. **指望一次生成搞定**——必须有 lint + 重写循环
5. **依赖模型的"自我承诺"**——模型说"我会注意"不代表生成时能做到

## 这个项目的设计取舍

基于以上研究，仓库的分层是：

| 层 | 作用 | 设计原则 |
|---|---|---|
| **Rule（always-on）** | 前置硬禁令 + 自检 | 最多 5 条，用 FORBIDDEN，末尾自检指令 |
| **SKILL.md** | 完整模式说明、替换方法、场景调整 | 按需触发，可以详细 |
| **references/** | 完整黑名单、案例库 | 备查，不在生成时直接 prime |
| **lint + rewrite** | 工程兜底 | 把规则没拦住的留给确定性脚本 |

**lint 不是 nice-to-have，是这个项目能不能真正生效的关键。** 因为模型权重层的模式坍塌没法靠 prompt 修，只能靠生成后的检测和重写补救。
